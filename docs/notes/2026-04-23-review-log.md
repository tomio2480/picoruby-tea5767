# 設計書レビューログ（2026-04-23 全面改訂時）

## 背景

SSD1306 廃止・Web Serial + Ruby.wasm 採用の方針決定に伴い，`picoruby-tea5767-plan/` 配下の設計書を全面改訂した．改訂時に自己検証で 6 点の疑問点が挙がったため，ユーザーにレビューを依頼．

## 指摘と判断

### 1. スコープ外の再整理

- **指摘**：旧版は「音声再生しない・RDS 復号しない・電池駆動しない」の 3 点．新版で「スタンドアロン動作しない（表示と操作は PC 側）」を追加したが，RubyKaigi デモ前提として適切か？
- **判断**：そのままでよい．
- **対応**：修正なし．

### 1b. 追記：76.0 MHz の PLL 期待値は 9304（9305 への「訂正」は誤り）

- **訂正**：初回レビュー時に「`pll_for(76_000_000) == 9304` を 9305 に直すべき」と指摘したが， **これは計算ミスだった** ．
- Phase 4 の `tea5767.rb` TDD 着手時に minitest が失敗して発覚．
- 正しい計算：`4 × (76_000_000 + 225_000) / 32_768 = 304_900_000 / 32_768`
  - 9_304 × 32_768 = 304,873,472
  - 9_305 × 32_768 = 304,906,240
  - 304,873,472 < 304,900,000 < 304,906,240 → 整数除算の結果は **9304**
- 初回の計算メモ「9305.419...」は割り算を誤っていた．正しくは「9304.809...」．
- 対応：
  - `firmware/spec/tea5767_test.rb` を 9304 に戻した
  - `picoruby-tea5767-plan/README.md` のテストケース例も 9304 に戻した
  - 本項を追記して経緯を残す．以後の類似指摘では値検算を 2 度行う

### 1c. 追記：選定基準を結果ベースで言語化（2026-04-29）

- gemini-code-assist のレビューを契機に，N の選定基準を **「実周波数と目標周波数の誤差を最小にする整数」** という結果ベースの表現で再定義した．
- 1b. では「整数除算で 9304 になる」という **計算手続きの結果** を採用していた．これは「どの操作で計算するか」を判断軸にしており，意図がコードや文書に直接表れていなかった．
- 上記基準のもとで 76.0 MHz を再評価すると次のとおり．
  - 連続解は 9304.81．
  - 9304 を選ぶと 75.9934 MHz（誤差 −6.6 kHz）．
  - 9305 を選ぶと 76.0016 MHz（誤差 +1.6 kHz）．
  - 基準にもとづき 9305 を採用．
- 82.5 MHz は連続解が整数に十分近いため floor / round いずれでも 10098．基準には影響しない．
- 実装では `(4.0 * (freq_hz + IF_FREQ_HZ) / XTAL_FREQ_HZ).round` と書き，コメントで **「目標誤差を最小にする整数を返す」** と明示する．`.round` はこの基準を線形関係下で実現する手段であり，丸めという操作それ自体が出発点ではない．
- 影響範囲：
  - `picoruby-tea5767-plan/README.md` の式・コードスケルトン・テスト例（9305）は本 PR で更新済．
  - `firmware/spec/tea5767_test.rb` および `firmware/lib/tea5767.rb` も同基準に揃える必要があるが，本 PR の対象外．後続のスタック PR（`feat/picoruby-api-and-final-cleanup` 系）で 9305 へ統一する．
- 教訓：コードと文書には **「どんな結果を求めるか」** を意図として書く．演算名（floor / round / ceil）は実装詳細であり，それだけを判断軸にすると今回のように方針がぶれる．

### 2. schematic.svg の 3V3 配線経路

- **指摘**：Pin 36 → TEA5767 VCC を「Pico 右側から出して TEA5767 左側に入る」形にしたが，視覚的に遠回りに見えるかも．
- **判断**：今のまま．ただしすべて重ならないこと．
- **検算**：4 系統（SDA y=230 / SCL y=254 / GND y=278 / 3V3 y=302）は 24 px 以上離れて重なりなし．左側の垂直迂回線（SDA x=54 / SCL x=40 / GND x=26）も 14 px 間隔で独立．ピンラベル領域（x ≥ 94）と配線領域（x ≤ 74）のセパレーションも成立．→ 問題なし．
- **対応**：修正なし．

### 3. kicad_sch の `USB_CDC_TO_BROWSER` ラベル

- **指摘**：KiCad の label は配線結線のセマンティクスを持つ．「USB は PC につながっている」ことを示すだけの装飾ラベルは KiCad の慣習として微妙か．
- **判断**：一旦このまま進めて，実際に KiCad で開いたときの違和感次第で判断する．
- **対応**：修正なし．

### 4. display_layout.svg の NHK-FM 日浦 83.6 MHz

- **指摘**：プリセット上は中継局として分けているが，視覚的には親局 87.0 MHz とほぼ同じ内容．会場（函館市街）での 83.6 MHz 受信強度次第では紛らわしい可能性．
- **判断**：把握．別エントリで残す方針を継続．
- **対応**：修正なし．

### 5. README の aggregator 節（Canvas 800 px なら集約不要）

- **指摘**：Canvas が 800 px なら 191 ch を 1 ch = 1 バー以上で描けるので集約は不要．`aggregator.rb` は「ウィンドウ縮小時の保険」として残す位置づけでよいか．
- **判断**：その方向で OK．
- **対応**：修正なし．

### 6. 法規面での送信側への注意書き

- **指摘**：v0.1 では完全に不要な内容（送信側 NS73M / NJM2035 等への言及）．削除と保持のどちらを採るか．
- **判断**：実装計画を順当に進めれば必要という添え書きをして残す．
- **対応**：README の法規面節末尾に次の段落を追加．

```
v0.1 は受信のみのため上記後半（送信側）は実装対象外だが，
本実装計画を順当に進めると v0.2 以降で送信側の実験が視野に入る．
そのタイミングで改めて参照できるようにここに書き残しておく．
```
